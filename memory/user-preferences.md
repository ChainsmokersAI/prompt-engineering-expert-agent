# 사용자 선호도

## 작업 환경

- **런타임**: Claude Code (주 작업 환경)
- **언어**: 한국어 (파일 내용, 대화 모두 한국어)


## 설계 철학

- **Task 기반 접근 선호**: E2E 워크플로우보다 개별 Task 단위 작업을 선호
- **각 전문가는 별도 프로젝트**: experts/ 하위에 독립적 프로젝트로 구성
- **실용주의**: 설계 문서만이 아닌, 실제 동작하는 파일을 함께 원함
- **구조적 분리**: 역할이 다른 기능은 별도 스킬로 분리하는 것을 선호 (create vs evolve)
- **공유 리소스 중복 방지**: 두 스킬이 공유하는 리소스는 별도 공유 디렉토리로

## 커뮤니케이션

- **질문 기반 설계**: 한 번에 완성본을 기대하기보다, 피드백 주고받으며 점진적으로 개선
- **구체적 피드백**: "좋아요"가 아닌, 무엇이 부족하고 어떻게 바꿔야 하는지 구체적으로 제시
- **공통 원칙 중시**: 모든 전문가가 지켜야 할 원칙을 명확히 정의하고 일관되게 적용하길 원함

## 중시하는 원칙

- **핵심 규칙 7가지**: 근거 기반 답변, 불확실성 명시, Memory 참조/기록 의무, Knowledge 참조/기록 의무, Skill/Subagent 지속적 개선, 비판적 사고, 기타(한글 작성/전문 용어 원문)
- **전문 용어 원문 유지**: CLAUDE.md 내에서 스킬 → skill, 서브에이전트 → subagent 등 전문 용어는 원문 표기 사용
- **Memory 시스템**: "정말 핵심"이라고 직접 강조. 모든 전문가에 필수. 작업 전 참조 + 작업 후 기록 의무
- **Memory 기록 자동화**: 사용자가 지시하지 않아도 매 작업 후 자동으로 Memory 업데이트 수행을 기대

## 문체

- **파일 수정 시 기존 문체 유지 필수**: 기존이 "~합니다"체이면 동일하게 유지, "~할 것", "~한다" 등으로 바꾸지 않음

## SKILL.md 작성 선호

- **핵심→절차→디테일→참조 순서**: 원칙을 먼저 제시하고, 절차 후 참조 링크를 하단에 정리
- **Skill/Subagent 지속적 개선 의무 중시**: 피드백 기반 즉시 개선 + 사용자 승인 필수 (CLAUDE.md 핵심 규칙 5번)
- **기능 맥락 파악 후 진행**: 인터뷰에서 기술적 질문(스타일, 도구)을 하지 않고, 리서치 후 제안

## 선호하지 않는 것

- E2E 단일 워크플로우 (부분 작업 불가)
- 근거 없는 설계 판단
- 긍정 편향적 답변
- 파일 수정 시 기존 문체를 무시하고 다른 어미로 작성하는 것
- 파일에 개인 정보 포함 (Git Push 대상 프로젝트이므로)
